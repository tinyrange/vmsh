Managed Guest Refactor Plan

Scope: cc VMM/runtime architecture used by vmsh. Written after reviewing the Linux, OpenBSD, FreeBSD, KVM, rootfs, guest init, and instance paths.

Assessment: the current architecture has good reusable low-level pieces, but the managed VM lifecycle is duplicated at the wrong layer. We are drifting toward one mini VMM orchestration per host and guest OS combination.

The fix should not be a large generic OS interface. The fix should be a small set of lifecycle abstractions: guest profile, root builder, boot planner, machine/device spec, host VMM, managed agent, and capability model.

Executive Direction

The right refactor is to make cc describe managed machines in two independent dimensions:

  • Guest OS policy: image names, release artifacts, root filesystem construction, boot inputs, init/agent behavior, and feature capabilities.
  • Host VMM execution: KVM, WHP, HVF, or sidecar translation of a machine spec into a running VM.

Today those dimensions are blended together. Linux on KVM, Linux on WHP, OpenBSD on KVM, and FreeBSD on KVM each accumulate their own startup and instance logic. That made sense while the only serious guest was Linux, but it becomes expensive as soon as BSD guests need the same managed features.

The product goal should be: adding NetBSD, illumos, Haiku, Plan 9, or another unusual guest should mostly mean adding a guest profile and filesystem/boot support. It should not mean copying a backend, guest agent protocol, network wiring, block-device session, and instance wrapper.

Architectural Diagnosis

The duplication is not primarily in virtio, block, filesystem, or netstack code. Those layers are already reasonably reusable. The duplication is in the control plane that turns a user request into a managed VM:

  1. Resolve an image reference such as alpine, @openbsd, or @freebsd.
  2. Fetch or materialize release artifacts.
  3. Build a root filesystem or root block device.
  4. Pick a kernel and boot layout.
  5. Attach virtio devices in a host-specific way.
  6. Wait for guest readiness markers.
  7. Speak the managed exec/copy/archive protocol.
  8. Expose the resulting instance to vmsh.

Those steps are currently scattered across backend files, rootfs packages, KVM session packages, and guest init binaries. That makes bugs show up as feature skew: a tty fix lands in one guest, a copy fix lands in another, DNS behavior works in Linux but not BSD, or package manager support depends on a one-off network path.

Key principle: the refactor should not erase OS differences. It should isolate them. OpenBSD, FreeBSD, and Linux genuinely boot differently and build roots differently. They should share lifecycle machinery, not pretend to share one boot implementation.

Current Shape

The code already has strong reusable foundations:

  • cc/internal/virtio owns block, net, console, rng, fs, vsock, and related device behavior.
  • cc/internal/netstack owns guest networking, DNS, TCP/UDP handling, and guest IP allocation.
  • cc/internal/fsimage, imagefs, and oci own filesystem construction, tar-backed roots, overlays, and image content.
  • cc/internal/vmruntime owns some shared runtime types, initramfs config, boot transcripts, markers, and exec framing.
  • cc/internal/amd64vm and cc/internal/arm64vm own arch layout and some device construction.

The issue is above those pieces. Startup orchestration, guest protocol handling, rootfs overlays, and instance feature wiring are repeated per OS.

Area Examples Problem

BSD KVM sessions openbsd_session_linux_amd64.go, freebsd_session_linux_amd64.go Same managed PC startup flow with OS-specific boot plan and minor device differences.

Risk if unchanged: every new operating system will need another copied guest init, another rootfs builder, another VMM session wrapper, another instance wrapper, and another set of bug fixes for exec, tty, copy, networking, and package manager behavior.

Where The Line Should Be

Layer Owns Must Not Own
Guest profile Image matching, release versions, artifact list, root policy, boot policy, agent capabilities. KVM loops, WHP/HVF details, vmsh command routing, generic exec protocol framing.
Root builder Turning artifacts into an imagefs.Directory, block backend, init payload, device nodes, and OS config files. Starting VMs, allocating guest IPs, deciding host virtualization backend.
Machine spec Memory, CPUs, boot inputs, serial ports, control channel, network, block devices, virtio devices. Image names, mirror URLs, package manager names, filesystem extraction rules.
Host VMM Translating machine spec into KVM/WHP/HVF/sidecar calls and returning a session. OpenBSD-vs-FreeBSD rootfs construction, Linux OCI behavior, vmsh UX policy.

vmsh User interaction, prompts, command routing, persistent shell state, display of capability errors. Guest rootfs internals, KVM device placement, guest protocol parsing.

Refactor Goals

Goal 1

One managed lifecycle. VM startup should follow one path from image reference to guest profile to root artifact to machine spec to host VMM session.

Goal 2

Explicit OS profiles. Linux, OpenBSD, FreeBSD, and future operating systems should plug into the lifecycle through focused interfaces.

Goal 3

Shared guest protocol. Managed exec/copy/archive/tty/signal behavior should live in one agent library with OS adapters.

Goal 4

Capability-driven features. Unsupported behavior should be reported from capabilities, not scattered as ad hoc errors.

These types should be boring data and narrow behavior. The point is not to make all operating systems identical; the point is to share the lifecycle while preserving OS-specific boot and rootfs logic.

Boundaries

Guest Profile

A guest profile owns OS-specific policy:

  • Which image names it handles, such as @openbsd and @freebsd.
  • How release artifacts are found and cached.
  • How the root filesystem is assembled.
  • Which kernel/boot artifacts are needed.
  • How networking is configured inside the rootfs.
  • Which managed features are supported.

It should not own KVM loops, PCI layout mechanics, common exec protocol code, or host-specific session lifecycle.

Root Artifact

type RootArtifact struct {
    Kernel      []byte
    Initrd      []byte
    RootBlock   virtio.BlockBackend
    RootFS      imagefs.Directory
    ExtraBlocks []virtio.BlockBackend
    Cleanup     func() error
    Metadata    map[string]string
}

Linux may use an initramfs plus virtio-fs or a block root image. OpenBSD and FreeBSD may use FFS/UFS block roots. Future operating systems may need FAT/ISO/ESP or multiple disks. The output should be a root artifact, not a host session.

Machine Spec

MachineSpec should describe the machine independently of the host backend. It should contain memory, CPU count, boot spec, serial/control setup, network device, block devices, and virtio devices. KVM/WHP/HVF translate that spec into platform calls.

Managed Agent

The guest agent protocol should be shared. The OS-specific init binaries should mostly wire platform adapters into the common protocol engine.

type AgentPlatform interface {
    StartProcess(context.Context, ExecRequest, io.Reader, io.Writer, io.Writer) (ManagedProcess, error)
    OpenPTY(cols, rows int) (PTY, error)
    Signal(name string, p ManagedProcess) error
    RootPath(rootDir, path string) (string, error)
    Shutdown(context.Context) error
}

The tar archive/extract code, request parsing, response framing, stdin handling, signal dispatch, resize handling, and marker writing should not be copied between Linux, OpenBSD, and FreeBSD.

Migration Plan

Phase -1: Stop The Drift

Purpose: avoid adding more copied operating-system paths while the refactor is in progress.

  • Any new managed feature must declare whether it is guest policy, host VMM behavior, agent protocol behavior, or vmsh UX behavior.
  • Do not add new OpenBSD-only or FreeBSD-only exec/copy/archive protocol handlers unless the difference is required by the OS API.
  • When a bug is found in tty, stdin, signal, archive, DNS, or control-channel behavior, first check whether the fix belongs in shared protocol/session code.
  • Keep a short conformance table for Linux, OpenBSD, FreeBSD, and Windows so feature skew is visible.

Exit criteria: reviewers can reject new duplicated lifecycle code unless the PR explicitly explains why it cannot be shared yet.

Phase 0: Lock Behavior With Characterization Tests

Purpose: make the refactor safe before moving code.

  • Add a managed guest conformance checklist covering boot, exec, streaming exec, TTY, stdin, signals, copy in, copy out, archive extract, DNS, package-manager smoke test, console history, and shutdown.
  • Add host-side unit tests for image matching and capability selection.
  • Record current OpenBSD and FreeBSD expected boot markers and fatal boot text behavior.
  • Keep existing rootfs/fsck validation tests as release gates for BSD filesystem work.

Exit criteria: tests exist for the behavior we are about to preserve, even if some are integration tests gated behind tags or local artifacts.

Phase 1: Extract Shared BSD Release Artifact Code

Purpose: remove low-risk duplication from openbsd/rootfs and freebsd/rootfs.

  • Create a small package for release artifacts, for example internal/managedos/release.
  • Move common cache directory normalization, mirror URL construction, fixture lookup, download, context-aware decompression, and atomic .tmp rename logic.
  • Keep compression-specific readers pluggable: gzip for OpenBSD sets, xz for FreeBSD sets.
  • Keep OS-specific artifact lists in each profile: bsd and base79.tgz for OpenBSD, kernel.txz and base.txz for FreeBSD.

Exit criteria: OpenBSD and FreeBSD still build the same rootfs, but shared artifact code owns the duplicated cache/decompress/download behavior.

Phase 2: Extract BSD Root Overlay Planning

Purpose: keep OS-specific files visible while making overlay construction reusable.

  • Introduce an OverlayPlan helper with files, device nodes, symlinks, and cleanup ownership.
  • Represent OS-specific network files as data: OpenBSD /etc/myname, /etc/hostname.vio0 or init script behavior; FreeBSD /etc/rc.conf.
  • Represent root mount metadata as data: OpenBSD /dev/sd0a / ffs; FreeBSD /dev/vtbd0 / ufs.
  • Keep special OpenBSD runtime library links and FreeBSD kernel extraction OS-specific.

Exit criteria: rootfs builders still read clearly as OpenBSD/FreeBSD code, but common overlay mechanics and cleanup are shared.

Phase 3: Share the Managed Guest Agent Protocol

Purpose: eliminate the most dangerous copied code.

  • Create a shared package for guest agent protocol code. Possible location: internal/managedguest/agent.
  • Move request structs, command loop, exec stream framing, sync, mkdir, write, tar extract/archive, marker writing, stdin handling, resize messages, signal names, and error responses.
  • Expose narrow OS adapters for process launch, pty, signal conversion, path handling, and any boot-time setup.
  • First migrate cmd/openbsd-init and cmd/freebsd-init. They are smaller and highly similar.
  • Migrate Linux init later. Linux has more mount/switch-root/module behavior, so only move its managed protocol once the BSD path proves the adapter shape.
Important: do not force Linux initramfs behavior into the BSD adapters. Share the protocol engine, not all boot setup.

Exit criteria: OpenBSD and FreeBSD init binaries compile from the same protocol engine and pass the same exec/copy/archive tests.

Phase 4: Collapse OpenBSD and FreeBSD KVM Managed Sessions

Purpose: replace copied KVM startup loops with one managed x86 PC session path.

  • Create StartManagedPCSession or similar under internal/hv/kvm.
  • Parameterize OS name, kernel, memory, root block, extra blocks, network device/stack, control port, boot planner, CPU mode installer, ready marker, and fatal text detector.
  • Represent PCI device layout as a small DevicePlan instead of hard-coding block/net placement in each OS session.
  • Keep OpenBSD and FreeBSD public wrapper functions initially. They should only normalize defaults and call the shared session builder.
type PCBootHooks struct {
    Prepare func(mem *GuestMemory, kernel []byte, opts BootOptions) (BootPlan, error)
    Enter   func(vm *VM, plan BootPlan) error
}

type ManagedPCConfig struct {
    GuestName   string
    Kernel      []byte
    MemoryMB    uint64
    Root        virtio.BlockBackend
    ExtraBlocks []virtio.BlockBackend
    Network     *NetworkSpec
    Boot        PCBootHooks
}

Exit criteria: OpenBSD and FreeBSD KVM files shrink to thin wrappers, and boot behavior remains unchanged.

Phase 5: Introduce Guest Profiles and Registry

Purpose: remove image-name special cases from Linux amd64 backend.

  • Create a registry that selects a GuestProfile for @openbsd, @freebsd, Linux OCI images, and blank Linux roots.
  • Move IsBuiltInImage, default version/arch/mirror, and runtime config into profiles.
  • Make Start, StartBlank, and Run call a common managed runtime path once the profile is selected.
  • Use profile capabilities to answer whether alternate image exec, shares, port forwarding, and snapshots are supported.

Exit criteria: backend_linux_amd64.go no longer imports OpenBSD/FreeBSD rootfs packages directly for image dispatch.

Phase 5.5: Make Network Identity A Shared Runtime Concern

Purpose: prevent each guest from inventing its own IP allocation and DNS behavior.

  • Keep guest IP allocation in one runtime/network layer, independent of Linux/OpenBSD/FreeBSD profile code.
  • Pass the assigned address, gateway, resolver, hostname, and search/domain data into root builders as structured data.
  • Make OpenBSD hostname.vio0, FreeBSD rc.conf, Linux init networking, and Windows networking consume the same network assignment model.
  • Keep DNS resolver behavior host-portable. Avoid Linux-only assumptions such as directly reading one host resolver format as the sole source of truth.

Exit criteria: two simultaneously running guests get distinct addresses and can communicate according to the virtual network policy, regardless of guest OS.

Phase 6: Extract Managed Instance

Purpose: stop copying host-side instance methods.

  • Create a reusable ManagedInstance that owns session, agent client, rootfs, network runtime, dmesg setting, and cleanup.
  • Implement common Exec, ExecStream, ConsoleHistory, Wait, Close, Flush, and NetworkIPv4.
  • Keep extension interfaces for Linux-only features: virtio-fs stats, dynamic shares, alternate image mounts, root snapshots, service proxy ports.
  • Turn unsupported behavior into capability errors generated from GuestCapabilities.

Exit criteria: Linux/OpenBSD/FreeBSD instances share common lifecycle and exec/copy behavior, with explicit capabilities for differences.

Phase 7: Split Host VMM From Guest Runtime

Purpose: make host backend files describe host virtualization, not every guest OS.

  • Introduce HostVMM implementations for KVM, WHP, HVF, and sidecar.
  • Move common request handling into a host-independent managed runtime service.
  • Let host VMMs translate MachineSpec to KVM/WHP/HVF details.
  • Keep arch-specific layout helpers in amd64vm and arm64vm, but avoid embedding guest OS policy in those packages.

Exit criteria: adding a new guest OS on Linux amd64 does not require editing the host backend except to register the profile.

Phase 8: Bring Windows And Sidecar Into The Same Shape

Purpose: prove the abstractions are host-neutral and not just BSD cleanup.

  • Map the Windows WHP path onto MachineSpec and the shared managed instance where possible.
  • Keep Windows-specific process and filesystem behavior in its guest profile and agent adapter.
  • Map sidecar instances onto the same capability model, even if the implementation remains a coordinator-backed host rather than an in-process VMM.
  • Use this phase to trim any KVM-shaped assumptions that accidentally leaked into generic runtime code.

Exit criteria: the common runtime can start at least one non-KVM host path or sidecar path without adding guest-specific cases to the host implementation.

Phase 9: Migrate Linux Init Protocol Last

Purpose: avoid destabilizing the oldest and most featureful guest while the shared agent engine is still settling.

  • First share the request schema and host-side managed agent client.
  • Move Linux command framing, stdin, signal, resize, archive, and copy behavior onto the shared guest agent engine after BSD proves the shape.
  • Keep Linux boot setup, module loading, initramfs mounting, virtio-fs setup, binfmt setup, and switch-root behavior in Linux-specific code.
  • Run the full Linux runtime test suite before and after each extraction.

Exit criteria: Linux, OpenBSD, and FreeBSD use one managed protocol implementation while preserving their distinct boot setup.

Capability Model

Capabilities should be explicit and testable. They should describe what the managed guest can do, not what happened to be implemented in a wrapper.

PR Stack

This should land as a sequence of small, mostly mechanical PRs. The order matters because each stage should reduce duplication without forcing the riskiest boot/session paths to move first.

PR Scope Expected Risk Required Verification
1 Conformance checklist, capability inventory, image matching tests. Low. go test ./internal/vm plus no behavior changes.
2 Shared artifact cache/download/decompression helper. Low to medium: release cache bugs affect boot time and local disk use. OpenBSD and FreeBSD rootfs tests, repeated build to prove cache reuse.
3 Shared root overlay/device/symlink planning. Medium: mistakes break /dev, network config, or runtime links. Rootfs tests plus guest boot smoke for OpenBSD and FreeBSD.
4 Shared BSD guestagent protocol engine. Medium: exec/copy/archive/tty regressions are user-visible. Cross-build BSD init binaries and run conformance exec/copy/archive tests.
5 Shared KVM managed PC session for BSD. High: boot setup, PCI placement, interrupt routing, control channel. KVM unit tests, OpenBSD boot, FreeBSD boot, fatal boot text tests.
6 Guest registry and profile dispatch. Medium: image names and vmsh launch behavior must stay compatible. Image dispatch tests and vmsh @openbsd/@freebsd smoke tests.
7 Reusable managed instance. High: affects exec, wait, close, console, network cleanup. Full Linux runtime tests plus BSD managed feature tests.
8 Host VMM split and MachineSpec. High: exposes hidden assumptions between guest policy and host backend. KVM Linux/BSD tests, then WHP/HVF compile and targeted runtime checks.
9 Linux guest init protocol migration. High: Linux is the broadest managed feature surface. go test ./..., Linux vmsh smoke, copy/archive/tty/signal/package tests.

Known Traps

  • TTY semantics: do not send an immediate stdin close for interactive TTY commands that have no host stdin stream yet. That can terminate shells before they become interactive.
  • Session lifetime: exec waits should surface VM/session termination explicitly. If the run loop exits with a network or device error, the managed exec path should report that root cause rather than hanging on a transcript marker.
  • IP allocation: address assignment must be centralized. Guest rootfs config should consume assigned network identity, not independently choose addresses.
  • DNS: the guest DNS server should support the records required by package managers, including SRV behavior where needed, and resolver discovery should be host-portable.
  • Artifact extraction: avoid extracting large OS sets to temporary host directories. Prefer seekable decompressed archives or lazy tar-backed roots so /tmp space does not determine whether a guest can boot.
  • Filesystem writes: BSD root block writers need explicit validation with guest fsck. A successful boot does not prove metadata correctness after writes.
  • Device placement: OpenBSD and FreeBSD may tolerate different PCI or virtio layouts. Shared session code must preserve working device placement as data.

Guardrails

  • Do not build a single large OperatingSystem interface with dozens of methods. Use small interfaces tied to lifecycle steps.
  • Do not force Linux initramfs and BSD real-root boot into the same implementation. Share the protocol and lifecycle, not the boot mechanics.
  • Do not hide OS-specific rootfs policy in generic helpers. OpenBSD and FreeBSD overlays should still be readable as OS-specific plans.
  • Do not change product behavior while extracting. Each phase should be reviewable as movement of responsibility plus focused tests.
  • Do not make host VMMs understand image names like @openbsd. Image selection belongs to the guest registry.
  • Do not treat missing features as incidental errors. They should be explicit capability failures.
  • Do not combine all phases into one large rewrite. The architecture improves when each PR removes one duplicated responsibility and leaves behavior measurable.

Implementation Checkpoint

The current refactor has enough of the foundation in place to show the intended shape, but it is not finished. The completed work should be treated as the shared substrate for the next passes, not as proof that new operating systems are now cheap by default.

Now Shared

  • OpenBSD and FreeBSD root builders use common release artifact, decompression, root plan, and root artifact helpers.
  • OpenBSD and FreeBSD init binaries use one guest-agent protocol engine for exec, stdin, signals, resize, copy-in, copy-out, archive, extract, and sync.
  • KVM OpenBSD and FreeBSD boot through a shared managed PC session path.
  • Linux, OpenBSD, and FreeBSD managed starts now carry explicit guest profiles and enter the shared runtime start service before host dispatch.
  • KVM, WHP, and HVF have host VMM adapters behind the shared managed host interface for managed Linux starts.
  • The wire protocol marker constants live in the platform-neutral protocol package, with shared guest-agent framing helpers and shared host-side exec event parsing.
  • Linux init uses shared guest-agent helpers for managed protocol framing and usage payload encoding, while keeping Linux-specific boot, pivot, credential, and readiness behavior local.
  • Linux init now has tested helpers for non-exec control dispatch, exec request validation/defaulting, and exec startup with active tracking, pending stdin-close, and initial stdin handling, preserving missing-command exit behavior while reducing inline command-loop policy.
  • Linux init now uses a tested systemd command-readiness gate helper instead of keeping readiness mutex state inside the command loop.
  • Linux init now uses tested guest-agent helpers for exec usage payload construction and signal-aware process exit-code normalization instead of carrying those managed accounting helpers locally.
  • Linux init now uses tested guest-agent helpers for managed exec user parsing, passwd/group root materialization, home/workdir creation, and file ownership adjustment instead of carrying that root metadata policy locally.
  • Linux init now uses a shared tested guest-agent exec reporter for begin, stdout, stderr, control, usage, timing, and exit marker emission while keeping Linux-specific phase choices local.
  • Linux init now has a tested managed-exec command construction helper for direct exec versus exec-pivot launch decisions, reducing inline process-launch policy inside the managed exec loop.

  • Linux init now has a tested managed-exec exit-code helper for cmd.Wait error normalization, including wait-delay suppression, non-exit failures, ordinary exit codes, and signal-derived exit codes.

Accepted Boundaries

  • Linux init keeps the actual Linux syscall bindings for exec-root pivot_root and exec-pivot credential application, while the pivot protocol, operation ordering, credential parsing, and process lifecycle behavior are named and tested helpers.
  • Windows and Darwin keep small platform-specific instance shells for host resources and VMM attachment, while shared code owns managed exec, file control, mount-state bookkeeping, delegated-resource forwarding, share conversion, network cleanup/fallback routing, snapshots, and lifecycle delegation.
  • The sidecar worker path keeps daemon launch wiring and served resource protocols sidecar-specific, while shared code owns managed-session adaptation, image-backed exec, blank-root passthrough decisions, resource composition, launch command construction, startup/hello validation, cleanup ordering, process wait/kill timeout handling, Unix listener setup, and boot/filesystem/network accept-once serving.
  • Capability checks cover managed file control requests, alternate-image exec, snapshots, shares, port-forward error reporting, conformance-gate alignment, and opt-in BSD boot contracts; routinely running full boot-level cross-OS conformance remains a CI/host-capability decision rather than a code-structure blocker.

Success Criteria

A new guest OS should require a new profile, rootfs/boot code for genuinely new formats, and a small guest-agent adapter. It should not require a copied backend, copied KVM session, copied exec protocol, copied instance wrapper, or copied artifact cache.

  • backend_linux_amd64.go is no longer the place where non-Linux OS support accumulates.
  • OpenBSD and FreeBSD use the same KVM managed PC session path.
  • OpenBSD and FreeBSD init binaries use the same guest protocol engine.
  • Managed feature differences are represented by capabilities.
  • Rootfs builders use common artifact and overlay helpers while keeping OS-specific policy explicit.
  • vmsh behavior for Linux, OpenBSD, and FreeBSD remains unchanged or improves under conformance tests.

Near-Term First Patch

The first implementation patch should be intentionally modest:

  1. Add a conformance checklist/test scaffold.
  2. Extract shared release artifact helpers from OpenBSD and FreeBSD rootfs packages.
  3. Keep public package APIs stable.
  4. Run existing rootfs tests and managed boot tests.

This creates momentum without touching the most fragile boot/session code first.

Final Target

The final architecture should make this sentence true:

For a new managed operating system, the developer writes a guest profile, root builder, boot planner, and small guest agent adapter; the existing managed runtime, host VMM, network, exec/copy/archive protocol, and vmsh integration do the rest.
