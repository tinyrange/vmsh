Standards for Terminal Interfaces
Joshua Scarsbrook - 26th June 2026
Overall Vision
Terminal Interfaces should be uncluttered and respect the time of the user. The user should know what’s going on at any given instant and they should always be able to take control away from the system.

A terminal interface should never appear frozen while it is responsible for the user waiting. It should acknowledge user action quickly, explain potentially unbounded waits, preserve user input, and provide a reliable path for the user to interrupt work and regain control.
Definitions
A “potentially unbounded wait” is any action a program takes where completion is not safely bounded by the program itself. This includes work whose completion depends on the filesystem, the network, a virtual filesystem, a remote machine, a virtual machine, another process, a device, the operating system scheduler, or any execution environment outside the program’s strict local interface.

The “strict local interface” is the carefully bounded and tested part of the program responsible for input polling, terminal rendering, signal handling, interrupt dispatch, spinner animation, and status reporting. This interface may be exempted from this standard only to the extent that it has been explicitly designed to remain responsive under the conditions this standard describes. The strict local interface must not perform potentially unbounded work on the input, rendering, signal, interrupt, or status-update path.

Everything outside the strict local interface must be treated as potentially unbounded, including system calls. This is especially important when the application may be running locally, remotely, inside a virtual machine, against a virtual filesystem, or over a network boundary.

A “responsiveness threshold” is the maximum time the program may allow to pass before giving the user visible feedback that their action has been acknowledged.

A “progress-detail threshold” is the maximum time the program may allow to pass before giving the user more explicit information about what is happening.

An “interrupt” is any user action that asks the application to stop waiting, stop work, return control, or move to a safe interactive state. Cancellation is one kind of interrupt. If an operation can be abandoned without committed side effects, the interrupt may be described as cancellation. If partial work may remain, the interface must describe the result as stopped, interrupted, timed out, failed, or partially complete rather than implying rollback.

For remote sessions, “acknowledgement” means a protocol acknowledgement where available. Where explicit acknowledgement is not available, acknowledgement means an observable remote response such as output, cursor movement, prompt update, screen update, or another transport-level liveness signal.
Presentation Modes
A terminal interface may present progress in different modes depending on the environment.

Dynamic interactive mode may use spinners, refreshed lines, progress bars, local status regions, and other dynamic terminal UI elements. This mode is appropriate when the relevant output stream is an interactive terminal capable of safe redraws.

Plain interactive mode should use short status messages, low-motion updates, elapsed time, and periodic plain-text feedback. This mode is appropriate when dynamic redraws would be disruptive or unreliable.

Non-interactive mode must avoid animations and dynamic terminal control sequences. This mode is appropriate when output is redirected, piped, captured by CI, written to logs, or attached to a dumb terminal. Human-facing progress and status should not corrupt machine-readable stdout.

Remote-session-health mode is locally rendered and describes transport or session state independently of the program running inside the remote session. This mode is appropriate when the terminal interface is connected to a remote session and the local interface can observe whether user input is being acknowledged.

Where the terminal supports safe out-of-band progress or session-health channels, the application may use them as a supplement to local progress feedback. These channels must not be required for correctness, and the application must still provide an in-terminal fallback.
Requirements
The core requirements are…

Any potentially unbounded wait must have a useful message ready to present to the user before the wait begins. The shorter and easier to parse the better.
The responsiveness threshold is 50ms. Only present the message if the wait time exceeds 50ms. This is not a completion budget; it is the maximum time before visible feedback is required. Some operations, such as a request to a local VM, might normally receive a response in 1ms, but could take 5 seconds if the VM is hanging. The message has to be ready in case that time is exceeded. Consider what happens if the application thinks a local file is being read but actually the file is stored on a network server or a virtual filesystem.
The progress-detail threshold is 500ms. Displaying a spinner with a message is better than nothing, but any time an action could reasonably be expected to take over 500ms the interface must be prepared to break down what is happening and provide progress from hard facts. For example if a file is being read then the application should use information like the file size, bytes read, bytes remaining, and current read speed to provide real progress and a current estimate based on observed throughput.
Progress detail should be specific enough to help the user decide whether to wait, interrupt, retry, or investigate. It should not expose implementation details that do not help the user understand the wait or choose an action.
Progress information must not use fake precision. The application must not hard code assumptions such as “this usually takes 100ms” and present those assumptions as progress. Progress may show bytes, records, files, stages, current operation, elapsed time, measured throughput, and estimated completion time when those values are grounded in real metrics. It must not show unsupported percentages, fake stages, or hard-coded estimates.
Determinate progress must be monotonic for the scope it represents. It must not move backward, restart, or reach 100% before the operation is complete. If the denominator changes, the interface must either explain the change or switch to stage-based or indeterminate progress.
When the total size or total amount of work is unknown, the interface must show the current operation, elapsed time, measured units completed, and measured throughput instead of pretending to know the finish time.
If an operation enters a new potentially unbounded wait, the status must update to the most specific accurate description of what the application is currently waiting for.
Potentially unbounded waits should use explicit timeouts where a timeout is semantically valid. A timeout does not replace progress feedback or interruptibility; it is an additional bound that prevents silent indefinite waiting.
Expect users to be typing while they are waiting. Unless a wait is interrupted this input must be displayed and buffered then passed along to the next step. Buffered input must be displayed exactly as accepted. The exceptions are password fields or terminal states where echo is disabled; in those cases input may be accepted without being echoed. This is expected behavior inside a terminal. If the user starts typing as an app is starting, they expect that input to be there when the app is ready.
If a wait is interrupted, buffered input associated with the interrupted wait must not be replayed unless the user explicitly confirms it or the input remains visibly present for editing.
Any potentially unbounded wait that captures the prompt or prevents normal user input must have an interrupt path active before the wait begins.
Any potentially unbounded wait must be interruptible, returning visible user control within 50ms. Blocking work must not run on the path required to render interruption and restore control. If cleanup needs to continue after interruption, it must either continue in the background or enter a visible interruptible cleanup state. If cleanup requires another potentially unbounded wait, that wait follows the same requirements including interruption.
Interruption must restore the terminal to a valid interactive state. This includes cooked/raw mode, echo, cursor visibility, alternate screen state, prompt state, and partial line rendering.
If an interrupted operation may have committed partial work, the interface must describe the resulting state honestly. It must not imply that work was rolled back unless rollback actually completed.
Progress UI should be as small and local as possible. It must not create more disruption than the wait itself. If rich progress would interfere with the surrounding terminal environment, such as tmux, neovim, a pager, a log stream, or a pasted input sequence, the application should degrade to the least disruptive useful signal: a short status message, a spinner beside the prompt, or a single periodically refreshed line. Preserving user control and explaining the wait are more important than displaying detailed progress.
Rich progress UI must only be used when the relevant output stream is an interactive terminal capable of safe dynamic updates. In non-interactive, redirected, dumb, or non-dynamic terminals, the application must degrade to plain textual status messages at a lower cadence. Human-facing progress and status should not corrupt machine-readable stdout.
Spinner animation should update at least twice per second while visible in a dynamic interactive terminal, unless the user or environment requests reduced motion or the terminal cannot safely redraw. In those cases, the interface must provide an equivalent low-motion status signal, such as a stable message with elapsed time or periodic plain-text updates.
Semantic status should update only when the underlying facts change.
If progress UI was shown, completion must leave the terminal in an understandable final state. On success, transient progress may be cleared if the result is obvious. On failure, interruption, timeout, or partial completion, the application must leave a durable plain-text status explaining what happened and what state the operation is in.
If the terminal interface is connected to a remote session, the local interface must monitor whether user input is being acknowledged by the remote side. If user input has not been acknowledged for over 1 second, the interface must show a minimal locally-rendered session-health indicator, such as a thin status strip, red progress bar, or other small UI element. This indicator must make clear that the wait is caused by the remote session or transport rather than necessarily by the program running inside the session. It should show useful status updates such as waiting for acknowledgement, reconnecting, disconnected, or input queued, and must not rely on the remote side to render, update, or acknowledge it.
Session-health indicators may use color, such as red for transport failure, but color must not be the only way state is communicated. The indicator must include textual or structural status such as waiting for acknowledgement, reconnecting, disconnected, or input queued, and must respect no-color and dumb-terminal environments.
In remote sessions, locally displayed input that has not yet been acknowledged by the remote side must remain distinguishable from remote-confirmed output when that distinction matters. The user should be able to tell whether input is queued, locally echoed, acknowledged, or rejected.
These requirements apply when the application is directly at fault for the user waiting. For example the application does not need to show a spinner if the user runs sleep 60, but it does need to show a spinner if the request to start that command takes over 50ms. This carve-out does not waive local session-health indicators for remote sessions.
An implementation must not satisfy these requirements by suppressing input, interruption, rendering, or status updates while potentially unbounded work is in progress.

These requirements are built with the idea that we can’t predict how long an action will take and we don’t know how fast our users' execution environments are. Even a local file read that appears instant on a modern high-end laptop could mean a ¼ second delay for a user with an older machine, a remote filesystem, a hanging VM, or a constrained runtime. Knowing what you're waiting for and having a grounded estimate of how long the wait may continue is always better than sitting at an application that appears frozen.
