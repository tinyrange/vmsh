BSD OCI Artifact Plan

Move built-in OpenBSD, FreeBSD, and NetBSD distribution inputs from ad hoc mirror downloads to versioned OCI artifacts published to GHCR, while continuing to build and overlay the vmsh guest init binary locally.

Goal

Publish BSD root filesystem and kernel inputs as content-addressed OCI artifacts under:

  • ghcr.io/tinyrange/cc-openbsd:<version>
  • ghcr.io/tinyrange/cc-freebsd:<version>
  • ghcr.io/tinyrange/cc-netbsd:<version>

The vmsh runtime should pull these artifacts, verify the metadata, load the kernel bytes, read the rootfs tree, apply the existing runtime overlays, and build the bootable BSD filesystem image exactly as it does today.

Key decision: OCI provides the pristine BSD rootfs and kernel. vmsh still builds and embeds the current guest init binary locally, then overlays init, network config, device nodes, fstab, hosts, resolver config, and any other runtime-specific files.

Why Change

Reliability

BSD mirror structure, availability, and latency become less visible to users. GHCR gives us a single distribution path and a cacheable registry API.

Reproducibility

Artifacts become digest-addressed. A tag can point at a tested manifest, while callers may pin exact digests for deterministic boot inputs.

Simpler Runtime

The runtime no longer needs per-BSD mirror URL construction, decompression, and release-set fetching as the default path.

Artifact Shape

Use a normal OCI manifest or image manifest rather than placing a single opaque archive inside an image. The rootfs and kernel should be separately identifiable.

Recommended Layers

  • Root filesystem layer: application/vnd.oci.image.layer.v1.tar+gzip
  • Kernel layer or blob: application/vnd.tinyrange.bsd.kernel.v1
  • Optional metadata layer: application/vnd.tinyrange.bsd.metadata.v1+json

Recommended Annotations

{
  "org.opencontainers.image.source": "https://github.com/tinyrange/vmsh",
  "org.opencontainers.image.version": "7.9",
  "io.tinyrange.bsd.family": "openbsd",
  "io.tinyrange.bsd.version": "7.9",
  "io.tinyrange.bsd.guest_arch": "amd64",
  "io.tinyrange.bsd.kernel.media_type": "application/vnd.tinyrange.bsd.kernel.v1",
  "io.tinyrange.bsd.rootfs.format": "tar+gzip",
  "io.tinyrange.bsd.source.mirror": "https://mirror.example.invalid/pub/OpenBSD",
  "io.tinyrange.bsd.source.checksums": "SHA256"
}

The annotations should make it possible to reject the artifact early if the family, version, architecture, or kernel media type does not match what the selected built-in guest profile expects.

Runtime Integration

Keep the current managed BSD boundary: callers ask for a rootartifact.Artifact containing kernel bytes, a root block backend, an imagefs.Directory, metadata, and cleanup.

  1. Add an OCI-backed BSD artifact provider behind the existing BSDDefinition.BuildArtifact hook.
  2. Pull ghcr.io/tinyrange/cc-<bsd>:<version> into the existing OCI store or a small adjacent artifact cache.
  3. Resolve the rootfs layer as an imagefs.Directory.
  4. Resolve the kernel layer into raw kernel bytes.
  5. Build the guest init binary locally for the active guest architecture.
  6. Apply the existing per-BSD overlays for init, fstab, network, hosts, resolver config, device nodes, and runtime defaults.
  7. Build the FFS/UFS root block image and boot with the existing KVM, HVF, or WHP managed BSD path.

Local Utility

The initial implementation lives at cc/cmd/bsdoci. It downloads or reads BSD release inputs, normalizes the rootfs to a gzip OCI layer, extracts the raw kernel as a separate custom layer, writes an OCI image layout, and can push the manifest and blobs to an OCI registry.

Build an OCI layout from upstream sources

cd cc
go run ./cmd/bsdoci \
  --family openbsd \
  --version 7.9 \
  --arch amd64 \
  --cache-dir ../.cache/bsdoci \
  --out ../build/bsdoci/openbsd-7.9-amd64

Push to a local registry

cd cc
go run ./cmd/bsdoci \
  --family openbsd \
  --version 7.9 \
  --arch amd64 \
  --cache-dir ../.cache/bsdoci \
  --push localhost:5000/tinyrange/cc-openbsd:7.9 \
  --plain-http

Dry-run a GHCR push

cd cc
go run ./cmd/bsdoci \
  --family openbsd \
  --version 7.9 \
  --arch amd64 \
  --cache-dir ../.cache/bsdoci \
  --push ghcr.io/tinyrange/cc-openbsd:7.9 \
  --dry-run

For fast local testing, pass --source-dir with files named like the upstream artifacts, such as bsd and base79.tgz for OpenBSD 7.9.

Guest Init Policy

Do not publish vmsh's live guest init binary inside these OCI artifacts as the runtime contract. The init binary is coupled to the vmsh host code and guest-agent protocol, so baking it into published BSD images would create version skew.

The OCI artifact should contain only the upstream-derived BSD rootfs and kernel inputs. vmsh should continue to compile the correct cc-openbsd-init, cc-freebsd-init, or cc-netbsd-init binary at runtime or from its local cache, then overlay that binary into the root filesystem.

Tagging

Start with stable version tags and digest pinning support.

  • ghcr.io/tinyrange/cc-openbsd:7.9
  • ghcr.io/tinyrange/cc-freebsd:15.1
  • ghcr.io/tinyrange/cc-netbsd:10.1

If a version needs republishing, prefer a new build metadata tag such as 7.9-r2 and update the default mapping intentionally. Users who need exact reproducibility should be able to pin by digest.

Multi-Arch

BSD guest architecture is not always the same concept as OCI platform architecture. The manifest should be explicit about guest architecture.

  • OpenBSD amd64 for Linux amd64 hosts.
  • OpenBSD arm64 for Linux arm64 and Darwin arm64 hosts.
  • FreeBSD amd64 and arm64 where supported.
  • NetBSD amd64 and evbarm-aarch64 where supported.

OCI platform fields can still be used for registry selection, but vmsh should verify io.tinyrange.bsd.guest_arch before booting.

Fallback And Rollout

  1. Build and publish OCI artifacts in CI from the same upstream release inputs currently downloaded at runtime.
  2. Add an OCI artifact provider and keep the current mirror provider as fallback.
  3. Switch built-in BSD defaults to GHCR artifacts.
  4. Emit clear progress messages: pulling artifact, reading kernel, building local init, applying overlay, building root block.
  5. Once stable, make mirror fallback opt-in or keep it as a recovery path for development builds.

Risks To Check

  • Confirm redistribution expectations for OpenBSD, FreeBSD, and NetBSD release artifacts.
  • Confirm GHCR public pull behavior and size limits for the largest BSD rootfs layers.
  • Verify the OCI store preserves file metadata needed by BSD rootfs builds: uid, gid, modes, symlinks, hardlinks, and device-node intent.
  • Decide whether custom kernel media types should be accepted by the existing OCI pull code or handled by a dedicated artifact reader.
  • Add digest verification and source checksum recording so the build pipeline is auditable.

Definition Of Done

  • @openbsd, @freebsd, and @netbsd boot from GHCR-backed artifacts by default.
  • The runtime still builds and overlays the current local guest init binary.
  • Digest-pinned artifacts boot successfully.
  • Mirror download remains available during rollout.
  • CI validates each published artifact by booting it and running a basic guest command through the managed control channel.

This plan intentionally keeps boot, network, and guest-agent behavior in the existing managed BSD runtime. The distribution mechanism changes; the runtime contract stays stable.
