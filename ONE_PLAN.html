One Name Plan

vmsh should make systems easy to switch between by giving every visible target exactly one unique name. VMs, SSH hosts, the host, and future image sources should all participate in the same user-facing namespace.

Core Rule

Switching should always be @name, where name is unique and tab-completable. There should be no user-facing need to remember whether a target is a VM, an SSH session, a default image VM, or a named instance derived from another source.

The visible name namespace should reject conflicts entirely. No shadowing, no precedence rules, and no behavior that depends on the current mode.

Desired User Model

@ubuntu
@alpine
@ssh ws1
@host

@hello --from ubuntu
@other --from ubuntu
@ubuntu2 --from ssh:ubuntu

@ubuntu selects or creates the default Ubuntu system. @hello --from ubuntu creates a second independent system named hello using Ubuntu as its source. After creation, switching is just @hello.

Terminology

Term Meaning Example
System A user-visible target that can be selected with @name. ubuntu, hello, ws1, host
Source The thing a system is derived from or connected to. builtin:ubuntu, library/alpine, ssh:ws1
Name The globally unique handle used for switching, completion, copy endpoints, stop, restart, and status. @hello, @copy @hello:/tmp/a @host:/tmp/a

Namespace Rules

  • Every visible system name is globally unique.
  • host is reserved for the local machine.
  • Builtins reserve their bare names: ubuntu, freebsd, openbsd.
  • Docker library shorthand reserves the bare name when the shorthand is used as a system or source.
  • Conflicts are errors at creation time, not ambiguous behavior at run time.
  • Typed source prefixes such as ssh: and library/ disambiguate sources, not selected systems.
  • Creation can be explicit; switching must be boring.
  • There is no precedence order between VMs, SSH systems, images, and aliases. A name either resolves exactly once or it is rejected.
  • Unqualified names are system names first. Image names are only sources during creation, or when a builtin/default image is being created for the first time.

Conflict Example

If ubuntu is reserved by the builtin Ubuntu image, then @ssh ubuntu should fail. The user must choose a unique visible name. 

vmsh: cannot create SSH system "ubuntu": that name is reserved by builtin image ubuntu
Use a unique name, for example: @ubuntu2 --from ssh:ubuntu

Creation Semantics

Command Meaning Conflict Behavior
@ubuntu Select or create the default system named ubuntu from builtin Ubuntu. Allowed because the builtin reserves that name for itself.
@alpine Select or create the default system named alpine from Docker library Alpine. Allowed unless another system already reserved alpine.
@hello --from ubuntu Create or select a named VM system hello derived from builtin Ubuntu. Error if hello is already any visible name.
@other --from library/ubuntu Create or select a named VM system from Docker Hub library Ubuntu, not the builtin Ubuntu alias. Error if other is already taken.
@ubuntu2 --from ssh:ubuntu Create or select an SSH system named ubuntu2 connected to SSH host ubuntu. Error if ubuntu2 is already taken.
@ssh ws1 Sugar for creating or selecting @ws1 --from ssh:ws1. Allowed only when ws1 is free. If it conflicts, require explicit naming with --from ssh:ws1.

Source Syntax

--from should accept typed source strings. Bare source names are permitted when they follow the default source rules. The source resolver should normalize sources before saving durable state, so a moving tag never becomes the only record of what was booted.

Source Type Notes
ubuntu builtin image Reserved bare name. Current special Ubuntu source.
freebsd builtin image Reserved bare name.
openbsd builtin image Reserved bare name.
alpine library image Shorthand for Docker library image library/alpine.
library/ubuntu library image Explicit Docker library image, useful when a bare name is reserved by a builtin.
ubuntu:latest image tag Accepted as a requested source, but saved records should resolve it to the exact immutable image digest.
ssh:ws1 ssh SSH source using the user's SSH config for ws1.

Copy Endpoints

Copy endpoints should use the same unique visible names. The day-to-day syntax should not require vm: or ssh: prefixes for already-created systems. 

@copy @host:/tmp/a @hello:/tmp/a
@copy @hello:/tmp/a @ubuntu:/tmp/a
@copy @ubuntu2:/tmp/a @host:/tmp/a

Typed prefixes still matter for sources at creation time. Once a system exists, its visible name is the endpoint.

Internal Shape

vmsh still needs backend IDs, instance IDs, and source metadata internally, but users should mostly see systems. 

System {
  Name      string // "hello"
  Kind      string // "host", "vm", "ssh"
  Source    string // "builtin:ubuntu", "library/alpine", "ssh:ws1"
  Isolated  bool
  BackendID string // hidden implementation detail
}

The invariant is that Name is globally unique. Source can repeat: hello and other may both be derived from builtin:ubuntu.

Source metadata should be canonical for durable state. If a system starts from a tag such as ubuntu:latest, saved images and durable records should store the resolved immutable image identity or digest, not only the moving tag. Human-facing diagnostics can still show the friendly source that was requested.

Command Behavior

Operation Proposed Form Notes
Switch @hello Selects the unique system named hello.
Create named system @hello --from ubuntu Creates a VM system if hello is not taken.
Create SSH system @laptop --from ssh:work-laptop Avoids conflicts between SSH host aliases and image names.
SSH sugar @ssh ws1 Equivalent to creating or selecting @ws1 --from ssh:ws1 only when ws1 is free.
Stop @stop hello Targets the unique visible system name.
Restart @restart hello Restarts the same source/config, not a different image.
Status tree @ps Shows hello from ubuntu, ubuntu2 from ssh:ubuntu, and active nesting.

Isolated Systems

Isolation remains useful, but it should be an attribute of a named system rather than a separate visible namespace. 

@scratch --from ubuntu --isolated
@scratch

Since names are globally unique, there should not be both a shared scratch and an isolated scratch. If a user wants both, they should choose two names: 

@dev --from ubuntu
@dev-iso --from ubuntu --isolated

Resolved Decisions

Question Decision Reason
Should @ssh ws1 remain? Yes, as sugar for @ws1 --from ssh:ws1 when ws1 is available. It preserves the familiar flow while keeping the global name invariant. Conflicts must error.
What does “record Docker library systems” mean? Once shorthand such as @alpine creates a system, alpine is reserved by that system until it is stopped/removed. The name must stay stable for switching, completion, copy endpoints, and status. Durable VM state should make the record durable too.
How should future configured source conflicts report? Prefer a direct owner-and-fix diagnostic. Errors should say who owns the name and show a valid --from form.
How should saved images record source identity? Normalize entirely. Save immutable resolved image identity, such as the exact digest behind ubuntu:latest. Saved images should be reproducible and should not silently change if an upstream tag moves.
Can isolated and non-isolated systems share a visible name? No. Isolation is a system attribute, not a separate namespace. @dev and @dev --isolated should not silently refer to different machines. Use @dev and @dev-iso.
Should legacy selectors remain? Yes, temporarily, but only as compatibility aliases. @vm:name, @ssh:name, and --vm can keep old scripts working while docs, prompts, completion, and errors teach the one-name model.

Cleanup Opportunities

  • Hide --vm from normal docs and completion once @name --from source covers common workflows.
  • Retire @vm:name as primary syntax; keep it temporarily as a compatibility alias.
  • Stop exposing backend IDs such as name-isolated in prompts and ordinary errors.
  • Make @ps show name from source instead of forcing users to parse VM IDs and image fields.
  • Make copy endpoint parsing prefer @name:path; reserve typed prefixes for compatibility and source creation.
  • Replace shared/isolated collision handling with one rule: one visible name, one system.
  • Make @stop, @restart, and @save take visible system names.
  • Move source normalization into one resolver so builtin, Docker library, SSH, and future sources use the same conflict checks.
  • Keep source identity and visible name separate in durable records so renames and saved images do not rewrite what the system was created from.
  • Make tab completion list only valid visible targets in switch position, and list source strings only after --from.
  • Prefer additive migration: introduce the system registry and compatibility aliases before removing old spelling.

Error Cases To Avoid

The dangerous behavior is silent guessing. The resolver should reject any command where a name could plausibly mean more than one visible system or source.

Case Desired Behavior Example Diagnostic
@ssh ubuntu when ubuntu is builtin Error and force a unique name. vmsh: name "ubuntu" is reserved by builtin image ubuntu. Use @ubuntu2 --from ssh:ubuntu
@ubuntu --from alpine Error because ubuntu is reserved by builtin Ubuntu. vmsh: name "ubuntu" is reserved by builtin image ubuntu
@hello --from alpine after hello exists from Ubuntu Error; do not retarget the existing system. vmsh: system "hello" already exists from builtin:ubuntu. Run @hello to switch to it or choose another name
@copy @ubuntu:/x ... when ubuntu is not exactly resolvable Error before copying. vmsh: copy endpoint "ubuntu" is ambiguous or not created; create a named system first
Bare Docker library name conflicts with a configured source Error and require a typed source. vmsh: name "redis" conflicts with configured source redis. Use @redis2 --from library/redis
@ssh ws1 after @ws1 --from ubuntu Error; do not replace or retarget the existing VM. vmsh: system "ws1" already exists from builtin:ubuntu. Use @ws1 to switch to it or choose @ws1-ssh --from ssh:ws1
@hello --from ubuntu:latest saved as an image Save the resolved digest, not the moving tag alone. source: docker://ubuntu@sha256:...
Completion after @ Offer visible systems and unambiguous creatable defaults, not every possible source spelling. ubuntu, host, ws1; source-only forms appear after --from.

Migration Plan

  1. Add a system registry with globally unique visible names.
  2. Represent current host, VM, and SSH contexts as systems internally.
  3. Add @name --from source creation while keeping current commands working.
  4. Make tab completion list visible system names and valid source names.
  5. Teach copy endpoints to prefer @name:path and keep @vm:/@ssh: as compatibility aliases for a while.
  6. Update @ps, @stop, @restart, and prompt display to use system names.
  7. Deprecate --vm once the new model covers all common workflows.
