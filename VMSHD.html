vmshd Design Plan

Introduce vmshd as the authenticated local daemon for vmsh. It should host the existing ccvm API and the new vmsh session API in one process, so VM runtime state and shell workflow state have a single lifecycle, one cache identity, and one local security boundary.

Goal

Replace the current direct vmsh to ccvm relationship with an embedded daemon mode that serves both the existing ccvm runtime API and vmsh-specific shell/session APIs from the same process.

Key decision: vmshd is not a proxy process that starts a separate ccvm daemon. The preferred local architecture is one daemon process that mounts the ccvm handlers and the vmshd handlers on the same authenticated listener. 
release binary
  vmsh client mode
  vmshd daemon mode
    ccvm runtime API
    vmsh session API
    auth middleware
    watchdog leases
    shell/session registry

The first implementation can keep most of the existing ccvm API surface intact. The meaningful change is ownership: vmshd becomes the process users connect to, the state file users discover, and the local authority for both VM operations and shell sessions.

Why Change

Unified View

vmsh currently knows about shell state and ccvm knows about VM state. A user asking "what is running?" needs one answer that includes VM instances, host shells, SSH sessions, background jobs, attached clients, and detached sessions.

Simpler Lifecycle

One daemon process means one startup path, one health check, one watchdog lease system, one shutdown operation, one cache directory, and one place to reason about compatibility.

Security Boundary

Every local request should authenticate. Adding that once at the shared daemon boundary is cleaner than teaching multiple local services to coordinate secrets.

Non-Goals

  • Do not redesign the VM runtime at the same time. Existing ccvm semantics should survive the first vmshd landing.
  • Do not require a network-visible daemon. The daemon is local-user infrastructure, not a remote service.
  • Do not make terminal bytes part of the correctness protocol. Shell status, cwd, job state, and session attachment state belong in structured daemon data.
  • Do not make user-facing prose a hard API contract. Tests should assert structured state, status codes, auth behavior, and command routing.

Process Model

The released vmsh binary should be able to run in client mode or daemon mode. This matches the current embedded ccvm release pattern, but the internal daemon mode should become vmshd rather than raw ccvm.

Mode Entrypoint Responsibility
Client vmsh Interactive line editor, terminal bridge, command parser, and session attachment client.
Daemon VMSH_INTERNAL_VMSHD=1 vmsh Authenticated local API, ccvm runtime handlers, vmsh session registry, and durable daemon state.
Compatibility VMSH_INTERNAL_CCVM=1 vmsh Temporary compatibility path while existing code still starts the embedded ccvm mode directly.

Same-Process ccvm

The ccvm daemon code should become a mountable component. Instead of only exposing a top-level ccvmd.Main, it should provide a function that returns an http.Handler or registers routes on a shared mux using an explicit runtime object. 

type Server struct {
    Runtime *ccvmd.Server
    Sessions *SessionRegistry
    Auth *TokenAuth
}

func (s *Server) Handler() http.Handler {
    mux := http.NewServeMux()
    ccvmd.RegisterHandlers(mux, s.Runtime)
    RegisterSessionHandlers(mux, s.Sessions)
    return s.Auth.Wrap(mux)
}

The exact type names should follow the cc tree, but the dependency direction matters: vmshd composes ccvm runtime handlers inside the same process. It does not shell out to a sibling daemon during normal local operation.

State Files

The client needs a small discovery file in the cache directory. The file should be private to the user and should identify the daemon, transport, auth token path, launch identity, and compatibility level. 

{
  "kind": "vmshd",
  "version": 1,
  "addr": "127.0.0.1:55433",
  "socket": "",
  "token_path": "/Users/joshua/Library/Caches/vmsh/ccprod/vmshd.token",
  "launch_key": "path=...",
  "api_version": "2026-06-25"
}
  • The state file should be written with mode 0600 on Unix platforms.
  • A stale, unreadable, unauthenticated, or incompatible state file should be removed before starting a new daemon.
  • The state file should not contain the token value unless there is no safer practical alternative.
  • Existing ccvm.json readers may be supported for a short compatibility window, but new vmsh clients should prefer vmshd.json.

Authentication

All daemon requests should authenticate, including health checks, capability checks, VM routes, session routes, event streams, and shutdown requests.

Key decision: Generate a fresh high-entropy bearer token when the daemon starts. Store it in a local filesystem path with private permissions. The client reads the token path from daemon state and sends Authorization: Bearer <token> on every request.

Token Lifecycle

  1. Daemon creates the cache directory with private permissions.
  2. Daemon generates at least 32 random bytes using crypto/rand.
  3. Daemon stores the encoded token at vmshd.token with mode 0600.
  4. Daemon writes vmshd.json with the token path and listener address.
  5. Client reads vmshd.json, then reads the token file.
  6. Client sends the token with every HTTP request and WebSocket upgrade.
  7. Daemon removes the token file on clean shutdown when possible.

Failure Rules

  • Missing token: 401 Unauthorized.
  • Malformed authorization header: 401 Unauthorized.
  • Wrong token: 403 Forbidden or 401 Unauthorized, chosen once and tested.
  • Token file has permissive mode on Unix: client refuses to use it and treats the daemon as unsafe.
  • State file refers to a missing token file: client removes stale state and starts a new daemon.

Transport Preference

A Unix domain socket is the best default on Unix and macOS when the existing client stack can support it cleanly. The socket directory gives a second local permission boundary. TCP loopback plus bearer token is still acceptable and preserves the current cc client shape.

Security boundary: Token auth protects against other local processes that can reach the listener but cannot read the user's private cache files. It is not a remote multi-user authorization system and should not be exposed beyond localhost or a private socket.

Session Registry

vmshd should own the authoritative list of shell sessions. A shell session is broader than a VM instance: it may include host shell state, selected VM context, SSH clients, persistent guest shells, background jobs, terminal attachments, and copy operations. 

Session {
  id: "sess_...",
  name: "main",
  owner_uid: 501,
  state: "attached" | "detached" | "closing",
  created_at: "...",
  updated_at: "...",
  selected_context: Context,
  attached_clients: [ClientAttachment],
  host_shells: [ShellHandle],
  guest_shells: [ShellHandle],
  ssh_shells: [ShellHandle],
  jobs: [JobSummary],
  copies: [CopySummary],
  vm_refs: [VMRef]
}

Session Properties

  • Sessions are named by users but addressed by stable daemon-generated IDs.
  • Multiple clients may attach when the terminal model supports it, but the first implementation can enforce one interactive attachment per session.
  • Detached sessions may keep shells and background jobs alive.
  • Session state should be observable without attaching to the terminal stream.
  • Sessions should carry enough structured metadata to implement @status, @jobs, and future session list views without scraping terminal output.

Unified Status

A single status request should answer the practical question: what is vmsh doing right now? 

GET /vmsh/sessions
GET /vmsh/sessions/{id}
GET /vmsh/jobs
GET /vmsh/status

The response should include session summaries, VM instance summaries, active streams, attached clients, and daemon metadata. Tests should assert structured fields rather than wording.

Terminal and Control Planes

vmshd should reinforce the design from SHELL.md: terminal bytes are opaque and control data is structured. The daemon registry makes that cleaner because command lifecycle and shell metadata can be sent to vmshd side channels instead of being parsed from visible output.

Plane Data Correctness Rule
Terminal PTY input/output, resize, interactive control bytes Forward as opaque bytes. Do not parse for command status.
Control Command lifecycle, cwd, exit status, selected context, jobs Send structured records through daemon APIs or private control channels.
Runtime VM start, exec, image pulls, port forwards, capabilities Use the mounted ccvm runtime handlers through the authenticated daemon.

API Shape

Keep ccvm routes stable where possible and add vmsh-specific routes under a clear prefix. That gives existing client code a path to move without changing every runtime operation at once.

Daemon Routes

  • GET /healthz checks authenticated daemon reachability.
  • GET /capabilities returns existing runtime capabilities.
  • POST /shutdown requests daemon shutdown.
  • POST /watchdog/lease keeps the daemon alive for active clients.

vmsh Routes

  • GET /vmsh/status returns daemon, runtime, and session summaries.
  • GET /vmsh/sessions lists shell sessions.
  • POST /vmsh/sessions creates a session.
  • GET /vmsh/sessions/{id} reads one session.
  • @sessions reads /vmsh/sessions and shows daemon-owned session resource counts.
  • @status reads /vmsh/status when attached to vmshd and shows daemon/session counts.
  • POST /vmsh/sessions/{id}/attach attaches an interactive client.
  • POST /vmsh/sessions/{id}/detach detaches a client.
  • POST /vmsh/sessions/{id}/jobs starts a daemon-owned host, VM, or host-origin SSH job.
  • DELETE /vmsh/sessions/{id}/jobs/{job} requests cancellation for a daemon-owned job.
  • DELETE /vmsh/sessions/{id} closes a session, marks daemon-owned running jobs as canceling, and removes owned resources from daemon views.
  • GET /vmsh/events streams daemon-level events.

Compatibility

Existing ccvm routes can remain at their current paths. If vmshd later needs versioned routes, add them without breaking the current local client all at once.

Client Behavior

The vmsh client should connect to vmshd by default. It should only fall back to direct ccvm mode behind an explicit compatibility path or while the refactor is in progress.

  1. Resolve cache identity and locate vmshd.json.
  2. Read daemon state and token path.
  3. Authenticate a health check.
  4. Validate launch key, API compatibility, and required routes.
  5. Reuse the daemon if compatible.
  6. Start embedded vmshd if no safe compatible daemon exists.
  7. Warn when a new daemon is started, stay quiet on reuse.
  8. Create or attach a shell session.

The new daemon warning should continue to come from structured startup state, not from text scraping or process output guessing.

Implementation Phases

Phase 1: Authenticated Daemon Foundation

  • Add token generation, token file writing, and auth middleware.
  • Extend daemon state with kind, token path, launch key, and API version.
  • Teach the client to read the token and authenticate every request.
  • Add tests for missing, wrong, and accepted token cases.

Phase 2: Same-Process vmshd Entrypoint

  • Add VMSH_INTERNAL_VMSHD daemon mode to the release binary.
  • Factor ccvm daemon code into a mountable handler/runtime component.
  • Start vmshd instead of embedded ccvm from vmsh.
  • Keep existing ccvm routes passing through the same listener.

Phase 3: Session Registry API

  • Add in-memory session registry with stable IDs and names.
  • Add /vmsh/sessions and /vmsh/status.
  • Move shell-owned session metadata into daemon-visible state.
  • Keep terminal byte forwarding in the client until attach/detach semantics are explicit.

Phase 4: Shell Attachment and Detached Sessions

  • Define attach protocol for interactive terminal clients.
  • Track attached clients and terminal size in session state.
  • Support detach without killing daemon-owned shells or jobs.
  • Add structured start and cancel controls for daemon-owned host, VM, and host-origin SSH jobs.
  • Expose enough state for @status and future UI views.

Phase 5: Cleanup and Compatibility Removal

  • Remove direct embedded ccvm startup from vmsh once vmshd is stable.
  • Delete compatibility state readers after a reasonable transition window.
  • Consolidate daemon lease and shutdown behavior under vmshd.

Testing Plan

Tests should focus on objective behavior: structured state, request routing, auth results, daemon reuse decisions, session records, and process lifecycle.

  • Unit test token generation and file permissions. On Unix, assert 0600.
  • Unit test auth middleware with missing, malformed, wrong, and correct bearer tokens.
  • Unit test daemon state read/write and stale state cleanup.
  • Unit test that ccvm routes and vmsh routes are both mounted behind the same auth middleware.
  • Unit test daemon reuse versus fresh daemon startup using structured callbacks.
  • Unit test session create/list/read/delete behavior.
  • Integration test that a locally started embedded vmshd can answer authenticated /healthz, /capabilities, and /vmsh/status.
Test discipline: Avoid asserting incidental UI wording. Exact assertions belong on JSON fields, status codes, route selection, file modes, token acceptance, session IDs, and daemon state transitions.

Open Questions

  • Should the first transport remain TCP loopback for minimum churn, or should Unix sockets land before session attachment work begins?
  • Should token rotation happen only on daemon restart, or should vmshd support explicit rotation while clients are attached?
  • How much session state should be durable across daemon restart versus in-memory only?
  • Should attached clients be limited to one writer, or should vmshd support multiple read-only observers early?
  • Which ccvm package boundaries need to move so vmshd can compose the runtime without importing a command package?

Done Criteria

  • A local vmsh client starts or reuses vmshd, not a separate ccvm daemon process.
  • The same daemon process serves existing ccvm runtime routes and new vmsh session routes.
  • Every request requires the generated local bearer token.
  • Daemon state points to a private token file and is validated before reuse.
  • /vmsh/status returns a unified structured view of daemon, session, and runtime state.
  • Reuse stays quiet; fresh daemon startup warns through structured startup state.
  • Focused unit and integration tests pass without making incidental prose a contract.
