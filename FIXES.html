Ubuntu Guest Compatibility Fixes

The current Ubuntu guest is useful as a lightweight native arm64 dev box, but it is not yet a boring full Ubuntu VM. The most plausible next step is to let vmsh boot Ubuntu with Ubuntu's own kernel and modules by default.

Proposed Default

@ubuntu should default to a real Ubuntu cloud root filesystem, systemd init, and an Ubuntu kernel/module set. Other images should keep the current lightweight kernel unless the user opts into the Ubuntu kernel with --kernel ubuntu.

The goal is no new requirement for ordinary users. The alias should pick the boring path automatically: @ubuntu means “boot the closest thing to a normal Ubuntu server VM that vmsh can provide.”

Highest Risk: Writable Root Filesystem

The reported hardlink behavior on virtiofs is the most dangerous issue because it may be silent. A clear unsupported feature like missing overlayfs fails loudly; incorrect hardlink semantics can corrupt package stores, caches, build systems, and deduplicated data structures.

Booting an Ubuntu kernel will not by itself make a virtiofs writable root behave like ext4. The long-term boring VM design should give Ubuntu a normal writable root filesystem, or at least move high-risk writable paths to a filesystem with POSIX-correct hardlinks.

Kernel Plan

  1. Add a kernel selector to VM start requests and shell parsing: --kernel default and --kernel ubuntu.
  2. Make @ubuntu imply --kernel ubuntu and --init unless explicitly overridden.
  3. For non-Ubuntu images, keep the current lightweight kernel unless --kernel ubuntu is requested.
  4. Download/cache the Ubuntu cloud kernel and matching module tree alongside the Ubuntu cloud rootfs.
  5. Teach ccvm boot construction to use the selected kernel, initrd/module payloads, and kernel command line.
  6. Extend guest init to load or expose expected device/kernel features: tun, overlay, netfilter modules, binfmt_misc, and any Ubuntu module dependencies.
  7. Add feature probes after boot and surface degraded compatibility clearly in diagnostics.

Shell UX Plan

Command Behavior
@ubuntu Ubuntu cloud rootfs, systemd, Ubuntu kernel by default.
@ubuntu --kernel default Explicit escape hatch to the current lightweight kernel.
@alpine Current lightweight kernel and image behavior.
@alpine --kernel ubuntu Boot Alpine userland with the Ubuntu kernel if the backend can satisfy it.
@ubuntu --isolated --memory 4g Still uses the boring Ubuntu default; isolation should not imply the lightweight kernel.

Tradeoffs

  • Ubuntu kernel boot will be larger to download and cache than the current tiny kernel path.
  • Boot time may increase, especially on first run and when module payloads are unpacked.
  • Supporting an Ubuntu kernel across arbitrary userlands is useful, but not guaranteed to make every image behave like Ubuntu.
  • The root filesystem issue needs a storage design fix; kernel selection alone should not be marketed as solving POSIX disk semantics.
  • More kernel features mean a larger compatibility surface and a bigger test matrix.

Validation Matrix

The Ubuntu kernel path should not be considered done until these probes pass or fail with documented limitations:

  • mount -t overlay with lower/upper/work/merged directories.
  • test -c /dev/net/tun and a minimal TUN open/ioctl probe.
  • nft list ruleset and basic iptables/nftables availability.
  • mount -t binfmt_misc binfmt_misc /proc/sys/fs/binfmt_misc.
  • Optional QEMU user registration and a tiny foreign-arch ELF execution test.
  • Hardlink and xattr probes on /, /root, /home, /tmp, and any proposed writable-root replacement.
  • apt update, apt install python3-pip build-essential, and package candidate checks for universe packages.
  • Systemd service listing, journald reads, time sync status, and long-lived clock drift checks.
  • Docker/podman smoke tests only after overlayfs, netfilter, cgroups, namespaces, and TUN expectations are explicit.

Open Design Questions

  • Should Ubuntu kernel artifacts come from the same cloud image release as the rootfs, or from Ubuntu package repositories?
  • Do we build a normal writable disk image for Ubuntu, or use an overlay/block layer over the cloud rootfs?
  • How do we cache kernel/module/rootfs artifacts by architecture and Ubuntu release without surprising disk usage?
  • Should --kernel ubuntu imply Ubuntu module loading only, or also Ubuntu-specific initramfs behavior?
  • What compatibility promise do we make for non-Ubuntu userlands booted with the Ubuntu kernel?
